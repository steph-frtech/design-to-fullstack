# Glossary

Canonical term definitions for the DTFS platform. Use these definitions verbatim in generated docs, comments, and agent instructions. Do not introduce synonyms.

See: [[ARCHITECTURE_OVERVIEW]] · [[CONTROL_PLANE_MODEL]] · [[DELTA_SPEC]] · [[EXECUTION_FLOW]]

---

## Terms

**Agent** — A Claude Code sub-agent defined in `.claude/agents/dtfs-*.md`; specialized for one DTFS concern (product analyst, spec writer, codegen, etc.); 19 agents total. See [[AGENTS_OVERVIEW]].

**Apply** — The action of executing a validated DeltaSpec against the Control Plane: persisting model changes to Postgres schema `dtfs`. Never applied without prior `validateDeltaSpec`. See [[DELTA_SPEC]].

**Asset** — A media or binary resource (image, font, file) attached to a project; stored with `storage`, `mimeType`, `sizeBytes`, `contentHash`, `originalName`. Currently absent from contract compilation and emitters (P1 gap).

**Assumption** — A documented hypothesis about user, context, or behaviour captured in the SpecKit; carries a risk level; linked to specs. See [[ASSUMPTIONS]].

**BackendContract** — Compiled artifact describing all backend API routes, auth methods, and policies for a project; input to the Hono emitter (`emit-hono.ts`). See [[BACKEND_CONTRACT]].

**Behavior** — A domain-level capability attached to an Entity or Resource; maps to OperationSteps at codegen time. See [[BEHAVIOR_EXPANSION]].

**ChangeSet** — A named group of Revisions; represents a logical unit of work; can be committed or reverted atomically. See [[CHANGESET_REVISION]].

**Client App** — A full-stack application generated by DTFS from a project's spec; runs in its own schema `gen_<slug>` and its own Docker container. _(target: Docker runtime not yet implemented.)_

**Client App DB** — The Postgres database for a specific generated app; schema `gen_<slug>`; created at deployment time. Never written to from the Control Plane. See [[CLIENT_DATABASE]].

**Codegen** — The full pipeline from `generateApp()` call through contract compilation → emitters → file writes → artifact tracking. See [[GENERATED_APP_OVERVIEW]].

**Contract** — One of the three compiled artifacts (BackendContract, FrontendContract, SharedContract) that mediate between the Control Plane spec and the code emitters. Must be valid before codegen runs. See [[CONTRACT_COMPILATION]].

**Control Plane** — The DTFS meta-platform itself; runs on Postgres schema `dtfs`; stores all definitions (specs, entities, policies, operations, contracts). Never confused with any generated app. See [[CONTROL_PLANE]].

**Control Plane DB** — The Postgres database running schema `dtfs`; stores all platform definitions; connection via `@prisma/adapter-pg`. See [[DATA_OWNERSHIP]].

**DeltaSpec** — The canonical change format fed to `apply_spec`; contains 21 typed buckets of `create[]`, `update[]`, and `delete[]` arrays. Never applied without prior validation. See [[DELTA_SPEC]].

**DeploymentTarget** — A record tracking where and how a GeneratedArtifact is deployed. See [[DEPLOYMENT_TARGETS]].

**DSL** — Domain-Specific Language. DTFS uses three DSLs: Expr (expressions), Operation (operation programs), Policy (access rules). All share the Expr AST — no free-form JSONata in new code.

**Emitter** — A code-generation function that produces files from a contract; e.g. `emit-hono.ts`, `emit-next.ts`, `emit-prisma.ts`, `emit-auth.ts`, `emit-sdk.ts`. See [[GENERATED_APP_OVERVIEW]].

**Expr** — The typed AST used for all expressions across the platform (5 variants: Literal, Ref, Call, Cond, Not); catalog of exactly 8 closed functions — no free-form JSONata. See [[EXPR_DSL]].

**FrontendContract** — Compiled artifact describing all screens, forms, fields, actions, and data bindings for a project; input to the Next.js emitter. See [[FRONTEND_CONTRACT]].

**GeneratedArtifact** — A file produced by codegen; tracked with path, contentHash, and a `protected` flag (currently hardcoded `false` — known P0 bug). See [[GENERATED_ARTIFACTS]].

**Guardrail** — An enforced check that prevents unsafe pipeline operations. DTFS has five defined guardrails; two are P0 (validate-before-apply, protect-manual-files) and are currently not enforced in the execution path. See [[AI_DO_NOT_BREAK]].

**MCP tool** — A tool registered on the DTFS MCP server (`backend/src/mcp.ts`); callable by AI agents; 102 tools registered (39 canonical + extras). See [[MCP_TOOLS]].

**OperationStep** — One step in an Operation DSL program; 10 supported kinds (query, mutation, call, condition, etc.). See [[OPERATION_DSL]].

**PlatformSpecProposal** — A structured proposal (Layer 5 output) mapping requirements to concrete platform concepts (Entity, Resource, Operation, etc.) before a DeltaSpec is created. See [[REQUIREMENTS]].

**PolicyRule** — A rule in a Policy, evaluated via Expr; 12 PolicyOp kinds; controls access and data filtering. Policy→middleware compilation is currently a stub pass-through. See [[POLICY_DSL]].

**ProductSpec** — Layer-1 artifact extracted from natural language input; captures purpose, personas, goals, and glossary for a project. See [[PRODUCT_SPEC]].

**Requirement** — A discrete, traceable need extracted from specs; linked to source (ProductSpec/ScreenSpec) and mapped to platform concepts via RequirementMapping. See [[REQUIREMENTS]].

**Revision** — A single recorded change to the Control Plane spec; belongs to a ChangeSet; enables point-in-time revert. See [[CHANGESET_REVISION]].

**RuntimeInstance** — A running instance of a Client App in a Docker container with its own database. _(target: not yet implemented.)_ See [[RUNTIME_INSTANCES]].

**RuntimeTarget** — Configuration record declaring the target stack for a project (e.g., DEFAULT = hono-next); drives which emitters are used during codegen. See [[CONTRACT_TO_CODEGEN_MAPPING]].

**ScreenSpec** — Layer-2 artifact describing a single UI screen: fields, actions, dataNeeds, and states. See [[PRODUCT_SPEC]].

**SharedContract** — Compiled artifact describing shared types, enums, and Zod schemas consumed by both backend and frontend emitters. See [[SHARED_CONTRACT]].

**Slash command** — A `.claude/commands/dtfs/*.md` file defining a reusable Claude Code workflow; 20 commands currently registered. See [[SKILLS_AND_COMMANDS]].

**Spec Kit** — The set of natural-language spec artifacts (ProductSpec, ScreenSpec, OpenQuestion, Assumption, SpecArtifact) stored as both DB records and disk files under `spec-kit/`; bidirectional sync via `spec-kit-sync.ts`. See [[SPEC_KIT_INTEGRATION]].
