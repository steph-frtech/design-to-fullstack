# DATABASE_GENERATION

L'emitter Prisma (`emitPrismaSchema`) lit `CodegenSpec.entities` directement (exception documentée — voir [[../04_Runtime_Contracts/CONTRACT_TO_CODEGEN_MAPPING]]) et produit `prisma/schema.prisma` pour l'application cliente. Ce schéma contient les modèles métier (depuis Entity/Attribute/Relation) et les 4 tables Better Auth (User, Session, Account, Verification).

Liens : [[../04_Runtime_Contracts/BACKEND_CONTRACT]] · [[CLIENT_DATABASE]] · [[BETTER_AUTH_GENERATION]] · [[GENERATED_ARTIFACTS]]

---

## Source of truth

`docs/CODEGEN.md` (section "Type Mapping") · `backend/src/codegen/emit-prisma.ts`

---

## Pourquoi emit-prisma lit le spec brut

`emit-prisma` lit `CodegenSpec.entities` plutôt que le `BackendContract`. Raison : le schéma Prisma doit refléter exactement les Entities et leurs Attributes Control Plane sans transformation contractuelle. Les types scalaires Prisma sont déterminés par `FieldType` (enum Control Plane), pas par des descripteurs de route. Cette exception est délibérée et documentée dans `CODEGEN.md`.

---

## Mapping FieldType → scalaire Prisma

| FieldType | Scalaire Prisma |
|---|---|
| TEXT, TEXTAREA, EMAIL, PASSWORD, RADIO, SELECT, FILE, RICHTEXT, COLOR, TIME, HIDDEN | `String` |
| NUMBER, RANGE | `Float` |
| DATE, DATETIME | `DateTime` |
| CHECKBOX | `Boolean` |
| MULTISELECT | `String[]` |
| CUSTOM | `Json` |

## Structure du schéma généré

```prisma
// Generated by dtfs codegen — do not edit manually
generator client {
  provider = "prisma-client-js"
  output   = "../generated/prisma"
}

datasource db {
  provider = "postgresql"
  // url fournie via @prisma/adapter-pg au runtime (Prisma 7 — pas de url dans datasource)
}

// ─── Modèles métier (depuis Entity/Attribute) ─────────────────────────────────

model Customer {
  id        String    @id @default(cuid())
  email     String    @unique
  name      String
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
  invoices  Invoice[]
}

// ─── Tables Better Auth ───────────────────────────────────────────────────────

model User {
  id            String    @id
  name          String
  email         String    @unique
  emailVerified Boolean
  createdAt     DateTime
  updatedAt     DateTime
  sessions      Session[]
  accounts      Account[]
  @@map("user")
}

model Session {
  id        String   @id
  userId    String
  token     String   @unique
  expiresAt DateTime
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  createdAt DateTime
  updatedAt DateTime
  @@map("session")
}

// … Account, Verification identiques au pattern Better Auth
```

## Relations

Les `EntityRelation` Control Plane sont traduites en relations Prisma :

| Relation.kind | Prisma |
|---|---|
| `ONE_TO_MANY` | `model A { bs B[] }` + `model B { aId String; a A @relation(...) }` |
| `MANY_TO_MANY` | table de jonction implicite ou explicite |
| `ONE_TO_ONE` | `@unique` sur la FK |

---

## Migrations

Le schéma Prisma généré est placé dans `prisma/schema.prisma` de l'app cliente. Les migrations sont gérées par l'app cliente elle-même (`prisma migrate deploy` au démarrage du conteneur `migrate`). Le DTFS meta-platform ne gère pas les migrations de l'app cliente.

---

## AI usage

Ne pas modifier `prisma/schema.prisma` dans l'app générée manuellement — le marquer `protected` si des personnalisations sont nécessaires. Pour ajouter des champs, modifier l'Entity dans le Control Plane et relancer le codegen.
