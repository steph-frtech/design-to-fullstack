# Runtime Contracts — Overview and Cadrage

> Status: **design doc** — no models or code exist yet for this layer. Implementation begins in Phase 25.
> References: `plan.md` L790–1016, L1929–2184.

---

## 1. Why This Layer Exists

The current pipeline stops at ChangeSet:

```
Natural description
→ ProductSpec → ScreenSpec → Clarification
→ PlatformSpec → DeltaSpec → ChangeSet → ProjectSpec
→ Codegen (directly emits Hono routes, Next pages, Prisma schema)
```

The codegen emitters (`emit-hono.ts`, `emit-next.ts`, `emit-prisma.ts`, `emit-operations.ts`) read `CodegenSpec` (derived directly from Prisma) and generate code. This creates tight coupling: changing Hono to Fastify, or Next to Remix, requires touching the emitters AND the model layer.

The contracts addendum inserts an intermediate representation:

```
ChangeSet → ProjectSpec (stable)
→ RuntimeTarget        ← which tech stack?
→ BackendContract      ← what does the backend expose?
→ FrontendContract     ← what does the frontend render?
→ SharedContract       ← what types cross the boundary?
→ Codegen Plan
→ Generated code
```

**The golden rule:** never generate code directly from Control Plane concepts. Always pass through the contracts.

```
BEFORE:   Entity / Operation / Screen → code
AFTER:    Entity / Operation → BackendContract → Hono code
          Screen / Form / Action → FrontendContract → Next code
          Entity / Operation / Policy → SharedContract → shared types
```

## 3. The Four New Models (doc form — not yet in schema.prisma)

### 3.1 RuntimeTarget

Declares the tech stack for a project. One per project (or named variants).

```
RuntimeTarget {
  id             String   — cuid
  projectId      String   — FK Project (cascade delete)
  name           String   — "default" | "hono-next" | …
  backend        Json     — { framework: "hono", version: "~4.12", runtime: "node", apiStyle: "rest" }
  frontend       Json     — { framework: "next", version: "16.x", router: "app", rendering: "server-components-first" }
  auth           Json     — { provider: "better-auth", basePath: "/api/auth" }
  database       Json     — { provider: "postgresql", orm: "prisma" }
  packageManager String?  — "pnpm" | "bun" | "npm"
  runtime        String?  — "node" | "bun" | "edge"
  config         Json?    — extra key-value overrides
  createdAt      DateTime
  updatedAt      DateTime
  UNIQUE(projectId, name)
}
```

Version targets:
- Hono: `~4.12.x`
- Next.js: `16.x` (App Router, Turbopack, React Compiler)
- Better Auth: latest stable (handler on `/api/auth/*`, framework-agnostic)
- PostgreSQL + Prisma 7 (`@prisma/adapter-pg`, no `url` in datasource)
- Zod: `^3.x`
- pnpm: `10.x` (workspace protocol)

### 3.2 BackendContract

The compiled description of what the backend must expose. Generated by `compileBackendContract(projectId)`.

```
BackendContract {
  id              String   — cuid
  projectId       String   — FK Project
  runtimeTargetId String?  — FK RuntimeTarget (optional, scoped)
  apiBasePath     String   — default "/api"
  routes          Json     — array of RouteDescriptor
  schemas         Json     — Zod/TypeScript schemas for entities + DTOs
  middlewares     Json?    — auth/policy middlewares to mount
  auth            Json?    — Better Auth config fragment
  errors          Json?    — error codes + shapes
  generatedFrom   Json?    — provenance: { entities, operations, policies, authMethods }
  createdAt       DateTime
  updatedAt       DateTime
}
```

**Generated from Control Plane concepts:**

| Control Plane concept | Becomes in BackendContract |
|-----------------------|---------------------------|
| `Entity` | Prisma model + Zod schema + TypeScript type + Repository + DTO + `/api/<entity>` resource |
| `Attribute` | DB column + Zod field + form field type + validation rule + input/output DTO |
| `Resource` | Family of Hono CRUD routes (GET/POST/PATCH/DELETE) filtered by `exposedOps` |
| `Operation` | Explicit Hono endpoint (`POST /api/operations/<name>`) with input schema, output schema, handler, policy checks, steps runner, error mapping |
| `Policy` | Hono middleware + permission check function |
| `AuthMethod` (SESSION) | Better Auth email/password config + `/api/auth/*` handler + session middleware |
| `Asset` | `POST /api/assets`, `GET /api/assets/:id`, `GET /api/assets/:id/raw` |
| `EventDefinition` | Typed event payload + emitter + test fixture |
| `Workflow` | Orchestration descriptor (planned — not yet implemented) |
| `Trigger` | Event → workflow link (planned) |

### 3.3 FrontendContract

The compiled description of what the Next.js frontend must render. Generated by `compileFrontendContract(projectId)`.

```
FrontendContract {
  id              String   — cuid
  projectId       String   — FK Project
  runtimeTargetId String?  — FK RuntimeTarget
  routes          Json     — Next.js route descriptors (path → page)
  pages           Json     — page configs (layout, auth-guard, data needs)
  layouts         Json?    — shared layout descriptors
  components      Json     — component stubs (kind: server|client|form|layout|data)
  forms           Json     — form + validation + submit action
  dataBindings    Json     — fetch / query / props mapping
  actions         Json     — button/link/submit/navigation actions
  authGuards      Json?    — which pages require session / role
  generatedFrom   Json?    — provenance: { screens, components, forms, fields, actions }
  createdAt       DateTime
  updatedAt       DateTime
}
```

**Generated from Control Plane concepts:**

| Control Plane concept | Becomes in FrontendContract |
|-----------------------|-----------------------------|
| `Screen` | `app/<route>/page.tsx` (or `app/(group)/<route>/page.tsx`) |
| `Component` | `components/generated/<ComponentName>.tsx` (server / client / form / layout / data) |
| `Form` | React form component + Zod validation schema + submit action + API client call + error/success state |
| `Field` | `<input>` / `<select>` / `<textarea>` / checkbox / date picker / asset upload / hidden / computed |
| `Action` | Button / link / form submit / server action / API mutation / navigation |
| `DataBinding` | Server-side fetch / client query / props mapping / loading state / empty state / error state |
| `Policy` | Frontend auth guard (hide unauthorized UI actions) |
| `Translation` | i18n strings wired to locale |
| `Theme` | Tailwind config tokens |
| `Asset` | Next `<Image>` / file rendering / upload component |

### 3.4 SharedContract

The bridge between backend and frontend — shared types, Zod schemas, API client surface. Generated by `compileSharedContract(projectId)`.

```
SharedContract {
  id          String   — cuid
  projectId   String   — FK Project
  types       Json     — TypeScript interfaces (Entity DTOs, API responses, errors)
  schemas     Json     — Zod schemas (input/output validation)
  apiClient   Json?    — API client function signatures (frontend → backend)
  errors      Json?    — Error code enum + shapes
  events      Json?    — Event payload types
  createdAt   DateTime
  updatedAt   DateTime
}
```

**Contents:**

| Item | Source |
|------|--------|
| DTOs | Entity attributes → TypeScript interfaces |
| Zod schemas | Entity attributes + Operation I/O → `z.object(...)` |
| API response types | Operation output schemas |
| API error types | Error codes from BackendContract |
| Auth session type | AuthMethod → Better Auth session shape |
| Role type | AppRole keys → TypeScript union |
| Event payload types | EventDefinition.payloadSchema → TypeScript interface |

## 4. The Compile Contracts Phase

Between `ChangeSet` and `Codegen`, a new mandatory phase:

```
DeltaSpec
→ ChangeSet (committed)
→ ProjectSpec (stable snapshot)
→ compileBackendContract(projectId)   → BackendContract row
→ compileFrontendContract(projectId)  → FrontendContract row
→ compileSharedContract(projectId)    → SharedContract row
→ validateContracts(projectId)        → gate: must pass before codegen
→ generateCode()                      → GeneratedArtifact rows
```

Contracts are stored in DB rows (not ephemeral) so they can be:
- inspected by agents before generating code
- diffed across versions
- explained to the user
- regenerated selectively if only one layer changes

---

## 5. Why Indirection Through Contracts

Without contracts, changing the tech stack means modifying emitters that know too much about both the spec model and the target framework. Example scenarios:

| Change | Without contracts | With contracts |
|--------|------------------|----------------|
| Hono → Fastify | Rewrite `emit-hono.ts` + fix all spec references | Write new Fastify emitter that reads `BackendContract` (same data) |
| Next → Remix | Rewrite `emit-next.ts` end-to-end | Write Remix emitter that reads `FrontendContract` |
| Prisma → Drizzle | Touch emitter + schema model | New DB emitter reads `BackendContract.schemas` |
| Better Auth → Clerk | Rewrite auth emission + middleware | New auth emitter reads `BackendContract.auth` |

The Control Plane (Entity, Operation, Screen, Policy…) never changes when switching frameworks.

## 6. Codegen Architecture After Contracts

Generated output lives under `generated-app/` (monorepo-style):

```
generated-app/
  apps/
    api/                      ← Hono backend (from BackendContract)
      src/
        index.ts
        auth.ts               ← Better Auth handler
        routes/
        operations/
        policies/
        repositories/
        middleware/
        assets/
        events/
    web/                      ← Next 16 app (from FrontendContract)
      app/
      components/generated/
      lib/
        api/                  ← typed SDK (from SharedContract.apiClient)
        auth/
        schemas/
  packages/
    shared/                   ← Shared types (from SharedContract)
      src/
        schemas/
        types/
        errors.ts
        api-contract.ts
  prisma/
    schema.prisma             ← from BackendContract.schemas (DB part)
    migrations/
  tests/
    api/
    e2e/
    contract/
```

**Recommended generation order:**
1. `prisma/schema.prisma` (database schema)
2. `packages/shared/` (shared types + Zod schemas)
3. `apps/api/auth.ts` (Better Auth runtime)
4. `apps/api/routes/` (Hono backend API)
5. `apps/web/` (Next 16 frontend)
6. `apps/web/lib/api/` (typed SDK client)
7. `tests/` (test scenarios)
